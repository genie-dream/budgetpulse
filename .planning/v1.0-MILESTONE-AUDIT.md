## Cross-Phase Integration

### Verified Wiring (18 connections)

| Flow | From → To | Status |
|------|-----------|--------|

### Integration Issues (Non-Blocking)

**Issue 1: BUDG-06 / No post-onboarding currency change**
`BudgetEditForm` has no currency field. `SettingsPage.handleSaveSettings` never calls `settingsStore.setCurrency(...)`. Currency is effectively immutable after first-run onboarding. The BUDG-06 requirement ("user can set currency") is satisfied at onboarding. No UI path to change it afterward.
*Severity: Medium — not a v1 blocker; currency change deferred to v2*
*Affected: BUDG-06, SETT-01*

**Issue 2: BUDG-05 / monthStartDay hardcoded to 1 in onboarding**
`OnboardingWizard.handleFinish` line 40 writes `monthStartDay: 1` unconditionally. No payday UI in onboarding steps. This is a documented design decision (CONTEXT.md): payday setting is Settings-only, post-onboarding. `BudgetEditForm` includes `monthStartDay` field. Requirement is met via Settings.
*Severity: Low — by design per CONTEXT.md; monthStartDay IS editable in Settings*
*Affected: BUDG-05*

**Issue 3: DASH-07 / Backdated transaction edge case**
`transactionStore.addTransaction` appends to the full in-memory array without period filtering. Dashboard totals include all in-memory transactions. A backdated transaction (via optional date picker on `/add`) can inflate `totalSpent` until next mount when Dexie re-queries with period filter.
*Severity: Low — affects only backdated transactions in the same session*
*Affected: DASH-07*

**Issue 4: Dual currency state (latent fragility)**
Dashboard/add/history read currency from `settingsStore.currency`. Analytics reads from `config?.currency`. Both are set identically at onboarding and have no mutation path, so no runtime divergence occurs in v1.
*Severity: Low — latent risk for future development only*
